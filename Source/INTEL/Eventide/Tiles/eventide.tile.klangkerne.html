Dolby Atmos & Spatial Audio

Dolby Atmos in Hardware

Atmos is an object-based spatial audio format where sounds are encoded as discrete objects with metadata (position, velocity, volume) rather than channel-locked mixed down stereo or 5.1. The Atmos renderer converts these objects into speaker feeds — traditionally a CPU-intensive task.

Klangkerne implements the full Atmos renderer in dedicated silicon, offloading the compute burden entirely from the CPU. The result: object-based spatial audio at cinema quality without CPU throttling, and with latency under 10ms — a massive improvement over software rendering (typically 50–150ms).

DTS:X & Beyond

Klangkerne also supports DTS:X object-based audio with its own unique feature set:

  • Dynamic height channel upmixing for stereo-to-immersive conversion
  • Headphone virtualization with custom HRTF profiles
  • Object audio passthrough (up to 32 discrete audio objects)
  • Real-time mix monitoring and object inspector
  • Full DTS:X creative metadata support

A signal of Eventide's media-first DNA: Alloy OS also ships with a significantly enhanced Dolby Vision 2 implementation (the metadata standard for HDR video color grading) — indicating that the OS platform takes end-to-end media production and consumption seriously, from image to sound.

Sonica™ Audio Interface

Sonica™ is Intel and Bang & Olufsen's hardware-direct audio runtime, purpose-built to expose Klangkerne's full capabilities to applications on Alloy OS. Named for the sonic precision and sonicintension at its core, Sonica is the bridge between user software and the silicon audio engine — enabling sub-10ms latency audio I/O, spatial parameter control, and real-time DSP injection without any OS scheduling overhead or external codec intermediation.

The Latency Advantage: Vertical Integration vs. Layered APIs

Traditional PC audio architectures rely on multi-layer software abstraction. WASAPI and DirectSound both impose a ~100ms minimum latency floor — not accounting for physical device-to-headphones transduction latency — due to:

WASAPI / DirectSound Path:
  • • OS kernel scheduling (10–30ms)
  • • Audio driver abstraction layer (10–20ms)
  • • Hardware buffer queuing (30–40ms)
  • • External codec processing (20–30ms)
  • • Total: ~100ms+ (not counting headphones)

Sonica eliminates every abstraction layer. By placing the audio engine directly on a dedicated tile with its own power domain, and providing a hardware-direct API that bypasses OS scheduling entirely, Sonica achieves:

Sonica Direct Path:
  • • Hardware-direct API call (<1ms)
  • • Ring-buffer hand-off (0–2ms)
  • • Klangkerne silicon processing (2–5ms)
  • • Analog output with no external codec (1–2ms)
  • • Total: ~4–10ms (silicon to analog)

Reaction Time Impact: For real-time audio applications (gaming audio feedback, live monitoring, spatial audio interaction), this 90ms latency reduction translates directly into tighter, more responsive audio-visual sync and user perception of system responsiveness. A user speaking into a microphone with headphone monitoring hears their own voice feedback within ~10ms instead of ~100ms — a difference perceived as instantaneous vs. noticeable delay.

Windows Compatibility: Sonica Adapter Driver

On Windows installations of Alloy OS, Sonica presents itself as a WinAPI-compatible audio driver with full WASAPI, DirectSound, and legacy MME support. Applications see a standard Windows audio device, but audio routes directly to Klangkerne hardware through Sonica's passthrough adapter layer — zero OS scheduling intervention, zero external codec coloration. Developers can use standard Windows audio libraries while transparently gaining Sonica's sub-10ms latency and B&O Beosonic DSP capabilities.

This compatibility mode still preserves most of Sonica's latency advantage compared to native WASAPI/DirectSound on other platforms, since the underlying Klangkerne hardware remains untouched and Sonica's ring-buffer handoff is orders of magnitude faster than OS-level audio scheduling.
